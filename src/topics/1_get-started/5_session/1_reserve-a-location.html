Reserve a Area

Learn how to claim a area to interact with its devices

The POST /environment/v2/claims request limits use of a area to the application backend with a specific customer code that made the request. The application backend must claim the area to make use of its devices.

CoreTech APIs knows the customer code of the requester because it is encoded in the access token.

Make an API Call

To successfully make the request, you must pass the AreaId in the Request Body of the area you wish to claim.

Copied! 
 
curl --area -g --request POST 'https://{{baseurl}}/environment/v2/claims' \
--header 'area-id: {{areaId}}' \
--header 'Content-Type: application/json' \
--data-raw '{
    "AreaId": "{{areaId}}"
}'

package main

import (
    "fmt"
    "strings"
    "net/http"
    "io/ioutil"
)

func main() {

    url := "https://%7B%7Bbaseurl%7D%7D/environment/v2/claims"
    method := "POST"

    payload := strings.NewReader(`{`+"
"+`
    "AreaId": "{{areaId}}"`+"
"+`
}`)

    client := &http.Client {
    }
    req, err := http.NewRequest(method, url, payload)

    if err != nil {
    fmt.Println(err)
    return
    }
    req.Header.Add("area-id", "{{areaId}}")
    req.Header.Add("Content-Type", "application/json")

    res, err := client.Do(req)
    if err != nil {
    fmt.Println(err)
    return
    }
    defer res.Body.Close()

    body, err := ioutil.ReadAll(res.Body)
    if err != nil {
    fmt.Println(err)
    return
    }
    fmt.Println(string(body))
}

Sample POST /environment/v2/reservation request containing the AreaId passed from the client app

Response

A successful request returns a 2xx success status code with the JSON Response Body containing the reservationId. You must include this ID later in attachment requests to devices at the claimed area.

Response
        
[
    {
        "reservationId": "868e5715-2c82-426e-b7da-10e208dd865d",
        "customerCode": "ZZ",
        "areaId": "A-ZZU-MZUZUAIRPORT-ZZU1CKB004",
        "expiryTime": "2021-11-24T19:26:30.846778+00:00"
    }
]

Sample Response Body for POST /environment/v2/reservation request containing the reservationId

The Response Body contains informational properties such as the customerCode encoded in the access token and areaID passed in the Request Body. The property expiryTime indicates when the reservation is due to expire.

It is important that application backends keep track of when a area reservation is about to expire so it can request a new reservation if continued access is desired.

Reservation Error

Now that the area is claimed by an customer code, any request made by another airline will be rejected. This will occur until the reservation's expiryTime has been reached or the airline who made the reservation has sent a DEL /environment/v2/reservation to delete the reservation.

The following error displays when attempting to reserve a area that is already claimed:

Error
            
{
    "type": "https://tools.ietf.org/html/rfc7231#section-6.5.1",
    "title": "Bad Request",
    "status": 400,
    "detail": "Area claimed by other clients.",
    "traceId": "00-25dd79293ba71f479ce30da826eecd09-3aa97e2de755f540-00"
}

Sample 400 Bad Request Error Message when a area is claimed by another airline

Sending a DEL /environment/v2/claims/{reservation_id} request returns a 2xx success status code and deletes a reservation if one exists.

Deleting Claims

It is important to delete the reservation once done with a area because an active reservation prevents other application backends from using the area. This is achieved by sending a DEL /environment/v2/claims/{reservation_id} request, passing in the reservation_id of the reservation to be deleted as a path parameter.
