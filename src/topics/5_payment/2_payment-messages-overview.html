Service Bus Messages

Learn about the most common messages and how your app should use them

The following is not an exhaustive list of all possible Service Bus Message Types. Only the most common ones encountered when sending Remit API requests are described.

PaymentDeviceStatus

DeviceStatus as Online

After you connect to a device, a PaymentDeviceStatus message is displayed on the Service Bus that should indicate the DeviceStatus is Online with the DeviceInUse property set to false.

Service Bus Message
            
{
    "DeviceStatus": "Online",
    "DeviceInUse": "false"
}

PaymentDeviceStatus Service Bus Message Body showing the payment device is Online and not currently 'In Use'

"DeviceInUse": "false" is the default value when a POST /remit/v2starttransaction request has not been made yet.

DeviceStatus Changes

If a device status changes from Online to Offline or PoweredOff, a PaymentDeviceStatus message is sent with an updated DeviceStatus.

The following sample message is returned when the DeviceStatus of payment devices changes to PoweredOff.

Service Bus Message
            
{
    "DeviceStatus": "PoweredOff",
    "DeviceInUse": "true"
}

PaymentDeviceStatus Service Bus Message Body showing the payment device is PoweredOff

Process PaymentResponse

When a POST /remit/v2starttransaction request is sent, a payment transaction is initiated and a ProcessRemittanceResponse message is displayed on the Service Bus.

In the Request Body, a relationId should have been passed to uniquely identify the transaction.

The events triggered by this request on the device are not determined by the Remit API since payment devices exist outside the CoreTech IoT. For a chip and pin device, most likely the consumer will be prompted to enter a PIN code.

PrintReceipt

Once the consumer's bank has authorized the payment and enough funds are confirmed to exist in the bank account, a receipt is triggered for the payment and a PrintReceipt message displays on the Service Bus.

The airline may choose to print the receipt or store its data. A POST /remit/v2billprinted request must be made to acknowledge either that the receipt has been printed or stored. The payment device waits for this acknowledgment before proceeding to the next step in the payment transaction.

VerifySignature

For some clients, once a receipt is printed, the consumer may be required to write their signature on the receipt. In this case, there will be a space on the receipt for the consumer to write their signature.

After the consumer writes their signature on the receipt, an airline attendant compares the signature on the receipt to the signature on the payment card to determine if they match.

The attendant confirms they have verified the consumer's signature by pressing a button in an airline app, for example. A POST /remit/v2verifysigned request is sent to the Remit API to acknowledge this.

EndTransaction

Once a payment transaction has completed, a EndTransaction message displays on the Service Bus. A POST /remit/v2completetransaction must be made to acknowledge that the transaction has completed.

PaymentTransactionTimeout

After a certain amount of time has elapsed since the last payment request was made, a PaymentTransactionTimeout will display on the Service Bus.

A new transaction must be initiated to by sending another POST /remit/v2starttransaction request.

A new relationId must passed in the Request Body to identify the new payment transaction. Passing the same relationId of the transaction that timed out will result in an error.
