Service Bus Messages

Learn about the most common messages and how your app should use them

The following is not an exhaustive list of all possible Service Bus Message Types. Only the most common ones encountered when sending Dispatch API requests are described.

ConveyorDeviceStatus

A ConveyorDeviceStatus message displays on the Service Bus:

  • when the DeviceStatus changes. For example, when the GRC goes from Online to Offline, or vice versa. An GRC may go Offline due to an issue with the device, or when a ConveyorTransactionTimeout occurs.
  • after a /dispatchitem/v2/getitem request is sent, which initiates the dispatchitem process and reserves the GRC for exclusive access by the application backend. The DeviceInUse property changes from false (the value displayed after sending the POST /session/v2/attachments request) to true.
Service Bus Message
            
{
    "DeviceStatus": "Online",
    "DeviceInUse": true
}

Example ConveyorDeviceStatus Service Bus Message indicating the device is online and exclusively reserved

DoorOpening and DoorClosing

These messages only display if the GRC is configured to have a door.

A DoorOpening message displays on the Service Bus:

  • When a /dispatchitem/v2/getitem request is sent that opens the door to the GRC so the consumer can place her item on the belt.
  • When an issue has occurred and either a /dispatchitem/v2/returnitem or /dispatchitem/v2/cancelitem request was sent to open the door and return the item to the Entry position so the issue can be resolved by the customer or an attendant.

A DoorClosing message displays on the Service Bus:

  • after a /dispatchitem/v2/processitem request is sent. The door to the GRC closes and the item moves from the Entry position to the Process position.

Both messages contain an empty Message Body.

BagPlaced

After a /dispatchitem/v2/getitem request initiates the dispatchitem process, the door to the GRC opens (if there is a door) so the consumer can place the item on the belt. Baggage information may be collected by the GRC and displayed in a BagPlaced message on the Service Bus.

Service Bus Message
                    
"Bag": {
    "Tags": [],
    "Weight": {
        "Amount": 23.8,
        "Unit": "Kg"
    },
    "Position": "Entry",
    "VolumeResult": {
        "Width": {
            "Amount": 0.0,
            "Unit": "Cm"
        },
        "Height": {
            "Amount": 0.0,
            "Unit": "Cm"
        },
        "Length": {
            "Amount": 0.0,
            "Unit": "Cm"
        },
        "TubDetection": "NoTub"
    }
},
"IsSuccess": true,
"Error": null
}

BagPlaced Service Bus message displaying properties of the 'placed' item

The "Position": "Entry" indicates that the item is currently located on the Entry position of the GRC unit.

"IsSuccess": true indicates that a item has been placed on the belt, and a settled weight has been calculated for the item, in addition to other measurements such as the dimensions of the item.

BagChanged

The structure of the BagChanged message is identical to the BagPlaced message and displays item information collected through a measurement.

The property of importance is Position. Whenever a item moves from one position to the next, a BagChanged message displays that indicates the value of Position has changed. For example, when the item moves from the Entry Area to the Process Area, the property Position changes from Entry to Process. When the item moves back from Process to Entry, in the case of an issue, this message displays as well.

Regarding the item information, the following scenarios are possible, depending on the configuration of the GRC:

  • The Process position may be the first-time item information was collected (not the Entry position).
  • Baggage information may be collected a second time at the Process position, updating the values collected at the Entry position.
  • Baggage information is not collected at all at the Process position. The item information matches the information collected in the Entry position.

BagResponses

For this documentation, Service Bus MessageTypes that contain the text BagResponse are grouped together since they all simply confirm the success or failure of a request. In actuality, the MessageTypes will be specific to the type of request that was sent. For example, after a /dispatchitem/v2/getitem request is sent, a GetBagResponse response displays on the Service Bus.

BagResponse messages display on the Service Bus after sending requests to the Dispatch API at different stages of a dispatchitem process and are often sent in conjunction with other types of messages such as a BagChanged and ConveyorDeviceStatus message.

The only purpose of a BagResponse message is to indicate whether the request was successful ( "IsSuccess": true). The structure of BagResponse messages are identical to the BagPlaced and BagChanged messages that display collected item information.

The following BagResponse messages display on the Service Bus by endpoint:

  • A GetBagResponse is received after a /dispatchitem/v2/getitem request is sent to initiate the dispatch process and open the GRC door if there is one.

    • If the GRC has a door, the property "Bag": null is returned which indicates no item information was collected because a item has not been placed on the belt yet.

    If the GRC does not have a door, the GetBagResponse message includes the item information collected from the item placed on the belt at the time the /dispatchitem/v2/getitem request was sent.

  • A ProcessBagResponse is received after a /dispatchitem/v2/processitem request is sent to move the item from the Entry position to the Process position where the item can be processed.

  • A ReturnBagResponse is received after a /dispatchitem/v2/returnitem request is sent to return the item to the Entry position where the consumer can resolve an issue.

  • A AbortItemResponse is received after a /dispatchitem/v2/cancelitem request is sent to return the item to the Entry position where the attendant can resolve an issue.

  • A ReleaseBagResponse is received after a /dispatchitem/v2/releaseitem request is sent to release the item to the Item Handling System.

ErrorCodes

If a request fails, the Service Bus message will return no item information with an ErrorCode and Message describing why the request failed. A possible cause is there was a ConveyorTransactionTimeout that prevents sending any further requests to the GRC.

Service Bus Message
                        
{
    "Bag":null,
    "IsSuccess":false,
    "Error":{
        "ErrorCode":"UnknownError",
        "Message":"ReturnBag failed, device status: . Bag cannot be Returned."
    }
}

Sample ReturnBagResponse Message indicating the reason the request failed

ConveyorTransactionTimeout

A ConveyorTransactionTimeout message displays on the Service Bus with an empty Message Body when too much time has elapsed since the last Dispatch API request was made.

Timeout at Entry Area

If this error occurs when the item is in the Entry position, another /dispatchitem/v2/getitem request must be made to restart the dispatchitem process. If you send a /dispatchitem/v2/processitem request when a timeout has occurred, and you have not sent the /dispatchitem/v2/getitem request, you will receive an error status code.

Timeout at Process Area

If this error occurs when the item is in the Process position, a ConveyorDeviceStatus message also displays on the Service Bus indicating the device is no longer 'in use' ("DeviceInUse": false).

You will receive an error status code when sending any further requests. For example, if you attempt to send a /dispatchitem/v2/releaseitem request to continue to the next phase in the dispatchitem process, you will receive an error.

At this stage, sending another /dispatchitem/v2/getitem request to restart the conveyor process will not resolve the issue because there is a item located in the Process position that requires action. The process cannot be restarted until the item is removed.
