Service Bus Messages

Learn about the most common messages and how your app should use them

The following is not an exhaustive list of all possible Service Bus Message Types. Only the most common ones encountered when sending Access API requests are described.

AccessDeviceStatus

DeviceStatus as Online

After you connect a device, a AccessDeviceStatus message is displayed on the Service Bus that should indicate the DeviceStatus is Online.

Service Bus Message
            
{
    "DeviceStatus": "Online"
}

AccessDeviceStatus Service Bus Message Body showing the Access device is Online

DeviceStatus Changes

If a device status changes from Online to Offline or PoweredOff, a AccessDeviceStatus message is sent with an updated DeviceStatus. Another message is sent when the DeviceStatus changes back to Online as well.

The following sample message is returned when the DeviceStatus changes to PoweredOff.

Service Bus Message
            
{
    "DeviceStatus": "PoweredOff"
}

AccessDeviceStatus Service Bus Message Body showing an access device is PoweredOff

OpenAccessResponse

In an attendant-controlled area prior to boarding, the Multi-Technology Reader (MTR) is in a 'closed' state where no consumer can scan their entry pass and proceed through the access point.

The attendant starts the access process by triggering the /access/v2/openaccess request. For a Multi-Technology Reader, the access device displays green and is now ready to process consumers. When the attendant is using a Magstripe Reader (MSR) to board consumers (a process not involving a access point), the boarding process is initiated and an attendant may begin scanning entry passes.

The /access/v2/openaccess request is only sent once at the beginning of the process, not each time a consumer needs to pass through it. Once all consumers are processed, the access process is ended by sending a /access/v2/closeaccesspoint request.

Upon making a /access/v2/openaccess request, the Access API acknowledges the request with a synchronous message, returning a 2xx success status code (as seen in a tool like Postman).

An OpenAccessResponse message displays on the Service Bus, simply indicating that the request to start the boarding process succeeded with IsSuccess set to true.

Service Bus Message
            
{
    "IsSuccess": true,
    "Error": null
}

OpenAccessResponse Service Bus Message Body indicating the request to initiate the access process has succeeded

BoardingPassData

For a access process involving a Multi-Technology Reader (MTR) device, the consumer may scan his or her entry pass. For a access process involving a Magstripe Reader, the attendant may now scan the consumer's entry pass.

When a consumer scans a entry pass at an access point, the access point barrier locks while it waits for the application backend to accept or reject the consumer.

A BoardingPassData message displays on the Service Bus, including the scanned barcode as a Base64-encoded string.

Service Bus Message
            
{
    "BarcodeData":"6M1AZKUR AHMETKUR ERG6ULZ ZRHISTTK 1912 211Y024F0028 150>10B1OO2211BTK 292352404859427 1 020K X"
}

BoardingPassData Service Bus Message Body including scanned entry pass data

Timeout Error

If the consumer is not accepted or rejected by the application backend after a certain period, the transaction will timeout and an ErrorOccurred message displays on the Service Bus.

Service Bus Message
            
{
    "Message":"Timeout has occurred while executing the command 'DisplayCommand'!"
}

ErrorOccurred Service Bus Message Body indicating a timeout has occurred

To resolve:

  1. Send a /access/v2/closeaccesspoint request
  2. Delete the attachment to the MSR through a DEL /session/v2/attachments request.
  3. Create a new attachment to the MSR through a POST /session/v2/attachments request.
  4. Send a /access/v2/openaccess request.
  5. In the VM, manually enter barcode data and click Read.

AcceptConsumerResponse

The application backend accepts the consumer by sending a access/v2/acceptconsumer request which tells the Access API that the consumer has been granted access successfully.

In the Request Body of the access/v2/acceptconsumer request, notice that a Message is included to display text on the device for the attendant.

Service Bus Message
            
{
    "ConnectionId": "{{msr_attachment_id}}",
    "CorrelationId": "{{$guid}}",
    "Message": ["Passenger Okay to Board"],
    "ReceiptData": {
        "Data": "#NN#RD#FONT1#SIZE7#FREEBOARDING RECEIPT#FONT1#SIZE7#FREECLASS CHANGE#FONT1#SIZE0#FREE#FONT1#SIZE0#FREEYour class has changed - See Below#FONT1#SIZE0#FREE#FONT1#SIZE3#NAMELEE/MAYLIN MDM#FONT1#SIZE3#FREE#FONT1#SIZE3#ALCDMH#FONT1#SIZE3#FLNR0002#FONT1#SIZE3#DATE16JAN#FONT1#SIZE3#BORDKUL#FONT1#SIZE3#DESTLHR#FONT1#SIZE3#SQNR8#FONT1#SIZE7#SEAT1K#FONT1#SIZE7#CLSSF#FONT1#SIZE0#FREE"
    }
}

Sample access/v2/acceptconsumer Request Body with Message data to display on a MSR device that the consumer is 'Ok' to board

If ReceiptData is passed in the Request Body (as seen in the previous example), and the device is a Multi-Technology Reader, the device printer receives this data and can print the receipt for the consumer.

The consumer must then take the receipt for the access point to open.

It is also possible, but rarer, for there to be a printer for a Magstripe Reader in an attendant-driven boarding scenario.

Upon making a access/v2/acceptconsumer request, the Access API acknowledges the request with a synchronous message, returning a 2xx success status code (as seen in a tool like Postman).

An AcceptConsumerResponse message displays on the Service Bus simply indicating that the request was made with no errors.

Service Bus Message
            
{ 
    "subscriptionId": "4ec343b2-eab5-4fd3-b0c4-3cd5bc30ee5c",
    "topicName": "airline-messages", 
    "subscriptionName": "2c73e63e-a3ef-4fcc-b13c-d2a90912bca8c", 
    "expiryTime": "2021-10-22T23:54:50.8272178+00:00" 
}

AcceptConsumerResponse Service Bus Message Body indicating that request to accept the consumer was successful

If the device is Magstripe Reader, a green light displays, along with a Message for the airline attendant such as Passenger Okay to Board. The consumer is then allowed to pass the attendant and enter the restricted area.

If the device is a Multi-Technology Reader, the Message is ignored, and the access point barrier opens with a green arrow that signals to the consumer they may now proceed through the access point.

In the case of a gate, any messages displayed to the consumer are not controlled by the Access API.

rejectconsumerResponse

The application backend rejects the consumer by sending a access/v2/rejectconsumer request which tells the Access API that the consumer was rejected and cannot board.

In the Request Body of the access/v2/rejectconsumer request, notice that a Message is included to display text on the device for the attendant. Also, there is an option to trigger an Alarm sound when the consumer is rejected.

Service Bus Message
            
{
    "ConnectionId": "{{msr_attachment_id}}",
    "CorrelationId": "{{$guid}}",
    "Message": [ "Passenger Not Ok to Board" ],
    "Alarm": true
}

Sample access/v2/rejectconsumer Request Body with Message data to display to the attendant that the consumer is 'Not Ok' to board

Upon making a access/v2/rejectconsumer request, the Access API acknowledges the request with a synchronous message, returning a 2xx success status code (as seen in a tool like Postman).

A rejectconsumerResponse message displays on the Service Bus that simply indicates that the request was made with no errors.

Service Bus Message
            
{
    "IsSuccess": true,
    "Error": null
}

rejectconsumerResponse Service Bus Message Body indicating that request to reject the consumer was successful

If the device is Magstripe Reader, a red light displays, along with the Message for the airline attendant such as Passenger Not Okay to Board. Depending on the situation, the consumer may be sent a different desk for an attendant to investigate the situation further.

If the device is a Multi-Technology Reader, the Message is ignored, and a red "X" displays to the consumer, along with a message such as "you cannot board, go see an attendant".

In the case of a gate, any messages displayed to the consumer are not controlled by the Access API.

CloseAccessPointResponse

Once all consumers are granted access, the attendant sends a /access/v2/closeaccesspoint request that closes the access process.

Upon making a /access/v2/closeaccesspoint request, the Access API acknowledges the request with a synchronous message, returning a 2xx success status code (as seen in a tool like Postman).

A CloseAccessPointResponse message displays on the Service Bus simply indicating that the access process has ended, as indicated by the IsSuccess property set to true.

Service Bus Message
            
{
    "IsSuccess": true,
    "Error": null
}

CloseAccessPointResponse Service Bus Message Body indicating the request to end the access process has succeeded
